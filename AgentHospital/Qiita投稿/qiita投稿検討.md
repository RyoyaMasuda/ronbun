## 2章 執筆メモ — 背景・なぜ Agent Hospital か

> **注意**：製品名・サービス名は記事に出さない。「病院向けのRAGチャットアプリ」等の一般表現にとどめる。

### 2-1. 自身の開発背景 — 病院向けRAGチャットアプリ

**書くこと**

- 現在、病院向けの **RAGチャットアプリ** を開発している
- 現状は **単純なRAG** 構成（ドキュメント検索 → プロンプトに載せて回答）
- 院内マニュアル・資料を参照しながら質問に答える、といった用途を想定

**トーン**

- 自社プロダクトの宣伝ではなく、「実務の延長で論文に触れた」という個人の学習・調査ログ
- 具体的な製品名・クライアント名・社名は書かない

---

### 2-2. エージェント実装を検討する理由

**書くこと**

- 現状のRAGチャットでも一定の用途には足りる
- ただし **今後の展開** として、エージェント的な振る舞い（計画・ツール利用・状態管理・複数ステップの処理）が入れられれば、より良い製品になるのでは、と考えた
- Agent Hospital は「医療ドメインでマルチエージェント + 経験蓄積」を体系化した例として参考になる

**つなぎ**

- 「だからエージェント全般の事例として Agent Hospital 論文を読んだ」
- 「読むだけでなく、小さく再現して体感した」→ Part 2 へ

---

### 2-3. RAGに縛られない拡張の考え方

**書くこと（重要）**

- RAGチャットアプリだからといって、**全部をRAGで解決する必要はない**
- RAGはあくまで **一つの部品**（知識検索・コンテキスト補完）
- それ以外のロジックを **別レイヤーとして組み込んでもよい**：
  - ワークフロー制御（グラフ / ステートマシン）
  - ツール呼び出し（ファイル読み書き、API、シミュレータ）
  - 反省・フィードバックループ
  - ルールベースの前処理・後処理

**Agent Hospital との接続**

- 論文の MedAgent-Zero も「RAG + シミュレーション + 反省」がセット
- 経験ベースの更新はRAG的だが、正誤判定は **シミュレータ（非RAG）**
- ミニ版再現でも、Naive RAG（ベクトルDB一括検索）にこだわらず **エージェント型**（ツールで記憶を読み書き）を選ぶ、という判断と一致する

**一文でまとめる例**

> RAGチャットを作っているからといって、設計はRAG中心に閉じる必要はない。Agent Hospital は、その良い反例だ。

---

### 2-4. 医療LLM（Med-PaLM等）の限界

**書くこと**

- Med-PaLM 等：膨大なテキストから**スクラッチ学習またはファインチューニング**で医学知識をモデルに組み込む
- USMLE 形式の問題では高い性能
- 一方で **病院ワークフロー全体**（問診→検査→診断→治療→経過観察）をモデル化するには不向き
- 知識はあるが「実践で試行錯誤して強くなる」仕組みがない

---

### 2-5. スクラッチ・ファインチューニング vs 実践シミュレーション

**書くこと**

| | スクラッチ / ファインチューニング（Med-PaLM等） | 実践シミュレーション（Agent Hospital） |
|---|---|---|
| 何をするか | 大規模コーパスで事前学習、またはドメイン特化の FT | 仮想病院で診察を繰り返す（LLM は凍結） |
| 強み | 広い医学知識・試験問題 | ワークフローに沿った経験の蓄積 |
| 弱み | 実践プロセスの再現 | シミュレータ設計への依存 |

- 自分のRAGチャットは前者に近い（資料検索 + 回答）
- Agent Hospital は後者の参考事例

---

### 2-6. この論文に辿り着いた理由

**書くこと（2章の締め）**

1. 病院向けRAGチャットを開発中
2. 次のステップとしてエージェント化を検討

4. Agent Hospital は医療 × マルチエージェント × 経験蓄積の具体例
5. 論文を読み、ミニ版を再現し、**RAGチャットへの適応**を整理することにした

## Part 2：ミニ版構築（`実装検討.txt` との対応）

### ワークフロー

```
Pre-Consult → Interview ⇄ Patient/Env → Reflection
```

| ノード | 種類 | 役割 |
|--------|------|------|
| Pre-Consult | Python 関数 | `read_global_lessons` で過去知見をロード |
| Interview | LLM エージェント | 問診・検査指示・診断 |
| Patient/Env | Python or LLM | 症状・検査結果を返す |
| Reflection | Python + LLM | 正解比較・反省・`update_global_lessons` |

### 記憶設計（エージェント型 RAG）

```
storage/
├── doctor/
│   └── global_lessons.json   # 共通ノート（教訓・反省）
└── patients/
    ├── patient_001.json      # 個別カルテ
    └── patient_002.json
```

- 完全上書きしない（Append-only + 論理削除 `is_active = False`）
- Naive RAG（ベクトル DB 一括検索）ではなく、LLM がツールで読み書き

## Part 3：RAGチャットアプリへの適応

> Part 3 の「適応」は **MedQA 再現や実臨床投入ではない**。2章で述べた病院向けRAGチャットに、Agent Hospital から何を持ち帰れるかを書く。

### 11章の構成案

**11-1. 論文・再現で得た設計パターン**

ミニ版を通じて体感した、RAGチャットにも転用しうる要素：

| パターン | Agent Hospital での例 | RAGチャットへの転用イメージ |
|---------|----------------------|---------------------------|
| 反省ループ | 誤診 → 原因分析 → ルール化 | 誤回答・不足回答のフィードバック蓄積 |
| 経験のファイル蓄積 | `global_lessons.json` | 運用ナレッジ・FAQ改善ログの構造化 |
| ワークフロー分離 | Pre-Consult / Interview / Reflection | 検索前処理・回答生成・事後検証のノード分割 |
| RAG以外のロジック | シミュレータによる正誤判定 | ルール検証、承認フロー、外部API |
| エージェント型記憶 | ツールで記憶を読み書き | 固定ベクトル検索に加えた能動的参照 |

**11-2. 現行RAGチャットへの具体的な適用案（要記入）**

- [ ] 例：問い合わせログから「よくある誤り」を Reflection 的に抽出

**11-3. あえて取り入れないもの**

- 診断・治療助言としての位置づけ

### 12章：適用の限界と次の一手

**12-1. 診断シミュレーション vs 院内資料QA**

| | Agent Hospital（ミニ版） | 病院RAGチャット |
|---|---|---|
| 目的 | 診察プロセスのシミュレーション | 院内資料に基づく Q&A |
| 正誤の基準 | シミュレータ上の治療結果 | 資料との整合・運用ルール |
| 記憶の中身 | 診断ルール・症例 | マニュアル・手順・過去の改善知 |

3. ツール呼び出しの拡張（RAG 以外の部品）
