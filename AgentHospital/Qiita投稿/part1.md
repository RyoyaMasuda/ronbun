## 2. 背景 — なぜ Agent Hospital か

Part 0 で触れたとおり、病院向け RAG チャットアプリを開発しています。本章では、なぜその延長として Agent Hospital 論文を深く読んだのかを整理します。

### 2-1. エージェント実装を検討する理由

**RAG の外側にも足せる設計**に興味を持ちました。複数ステップの処理、ツール呼び出し、状態管理、失敗から学ぶ仕組み——これらは RAG を置き換える話ではなく、**既存の RAG に並べて載せられるレイヤー**だと考えています。Agent Hospital は医療ドメインでの参考事例として目に留まりました。

### 2-2. RAG に加えて足せる拡張の考え方

RAG チャットのコアはそのまま活かしつつ、必要に応じて別ロジックを組み込めます。RAG は知識検索の**部品として中心に置ける**ままです。


| レイヤー   | 例                |
| ------ | ---------------- |
| RAG    | 資料検索、類似症例・ルールの取得 |
| ワークフロー | グラフ / ステートマシン    |
| ツール    | ファイル、API、シミュレータ  |
| 反省     | 失敗分析 → ルール化      |
| ルール    | 前処理・後処理、ガードレール   |


MedAgent-Zero は RAG 的な経験ベースを使いますが、正誤判定は**シミュレータ**が担います。**RAG を残したまま別レイヤーを足す**設計の参考になります。

### 2-3. 医療 LLM（Med-PaLM 等）の限界

論文が対比するのは、Med-PaLM のように**スクラッチから学習した、またはファインチューニングした**医療 LLM です。膨大なコーパスでモデル重みを更新し医学知識を組み込む一方、病院ワークフロー全体を直接モデル化する設計ではありません。

### 2-4. スクラッチ・ファインチューニング vs 実践シミュレーション


|       | スクラッチ / FT         | 実践シミュレーション      |
| ----- | ------------------ | --------------- |
| 何をするか | 大規模コーパスで事前学習 or FT | 仮想病院で診察（LLM 凍結） |
| 強み    | 広い医学知識・試験問題        | ワークフローに沿った経験蓄積  |
| 弱み    | 実践プロセスの再現          | シミュレータ設計への依存    |


院内資料 Q&A 向けの RAG チャットと Agent Hospital は**用途が違う**だけです。後者から学べるのは、ワークフローや経験蓄積の**設計思想**です。

### 2-5. この論文に辿り着いた理由

1. 病院向け RAG チャットを開発中
2. エージェント化を次のステップとして検討
3. RAG 以外のロジックも組み合わせられる設計を探していた
4. Agent Hospital = 医療 × マルチエージェント × 経験蓄積
5. 論文を読み、ミニ版を再現し、RAG チャットへの適応を整理（Part 2 / 3）

## 3. Agent Hospital の全体像

### 3-1. シミュラクラ（仮想病院）とは

**シミュラクラ**とは現実を模倣した仮想環境のこと。患者・看護師・医師がすべて LLM エージェントで、仮想時間は現実より数桁速く進みます。

> **図1**（論文 Figure 1）: Agent Hospital の全体像 — 16 の機能エリア

### 3-2. 登場するエージェント


| 種類        | 役割                   |
| --------- | -------------------- |
| 患者エージェント  | 症状を訴え、検査・治療結果で回復/未回復 |
| 看護師エージェント | トリアージ（症状を聞き取り、緊急度・診療科を振り分ける）、受付、検査結果の受け渡し   |
| 医師エージェント  | 問診、検査指示、診断、処方        |


32 診療科・300 以上の疾病をカバー。医師 42 名・看護師 4 名（論文設定）。

### 3-3. 診察〜治療の流れ

```
発病 → トリアージ（症状聞き取り・振り分け） → 受付 → 診察 → 医学的検査 → 診断・処方 → 薬の調剤 → 回復期
```

回復期のフィードバックが重要です。患者が回復したかどうかがシミュレータから返り、正誤判定と進化のトリガーになります。医師には勤務外の**読書**（医学書から知識統合）イベントもあります。

## 4. SEAL — 提案パラダイム

Agent Hospital の背後にある学習パラダイムが **SEAL**（**Simulacrum-based Evolutionary Agent Learning**）です。

### 4-1. シミュラクラ構築

**目的**：医師エージェントが学ぶための大量の医療データを、**人間によるアノテーションなし**に生成する。

- LLM と**医学知識ベース**（論文では百度健康百科など）を組み合わせる
- 疾病・症状・検査結果・治療後の経過などを仮想世界が自動生成
- 患者エージェントの属性（年齢、性別、病歴）も知識ベースに導かれて生成される

### 4-2. エージェント進化

**目的**：シミュラクラ内で、医師エージェントが成功・失敗の両方から経験を積み、能力を向上させる。

- 成功した診察 → **医療症例ベース**に蓄積
- 失敗した診察 → 反省して**診断ルール**を生成 → **経験ベース**に蓄積
- 次の診察で RAG 的に参照する

具体的な実装は **MedAgent-Zero**（5章）で述べます。

### 4-3. AlphaGo Zero / Smallville との共通点


|          | AlphaGo Zero | Smallville  | Agent Hospital（SEAL） |
| -------- | ------------ | ----------- | -------------------- |
| 人間の教師データ | 棋譜ナシ         | —           | 症例ラベルナシ              |
| データ生成    | 自己対戦         | エージェントの日常行動 | 仮想病院での診察             |
| 正誤判定     | 勝敗           | —           | 患者の回復/未回復            |
| 進化       | 対戦を重ねて強くなる   | 社会的行動の創発    | 診察を重ねて精度向上           |


Smallville が「仮想社会でエージェントが自律行動する」枠組みを示したのに対し、Agent Hospital は**病院ワークフローに特化し、医師エージェントの能力向上**を目的に設計されています。

## 5. MedAgent-Zero — 進化の仕組み

### 5-1. 「Zero」の意味（症例ラベルなし）

**Zero** = **手動でラベル付けされた症例データを使わない**（AlphaGo Zero と同じ発想）。


|       | 従来型（ラベル有り）               | MedAgent-Zero          |
| ----- | ------------------------ | ---------------------- |
| 学習データ | 「症状A + 検査B → 診断C」のラベル数万件 | シミュレーションが自動生成した患者      |
| 正誤    | 人間がラベルを教える               | 患者の回復/未回復で判定           |
| モデル   | ファインチューニングで重み更新          | **LLM は凍結**、経験はテキストで蓄積 |


「学習しない」の正確な意味は **5-5** と **7章** でも触れますが、ここでは**モデル重みは変えず、外部の経験ベースが成長する**と理解しておけば十分です。

### 5-2. 患者エージェントの自動生成

疾病名を入力すると、LLM + 医学知識ベースで次が順に生成されます。

1. **基本情報**（名前、性別、年齢）— 例：50歳以上は帯状疱疹に罹患しやすい → 年齢 55
2. **病歴** — 例：水痘の既往 → 帯状疱疹リスク
3. **症状リスト**
4. **医学的検査報告**
5. **品質管理エージェント**による整合性チェック

性別・年齢・国・疾病ごとに分布を制御できるため、特定コホートのシミュレーションも可能です。

> **図3**（論文 Figure 3）: 患者エージェントの生成フロー

### 5-3. 医師エージェントの診察と反省

医師エージェントは、患者の基本情報・病歴・症状だけを見て、**検査選択・診断・治療計画**を下します。

診察時の参照先は2つです。


| ストア         | 中身                        | いつ増えるか   |
| ----------- | ------------------------- | -------- |
| **医療症例ベース** | 過去の成功した Q&A ペア（質問 + 正解回答） | 治療成功時    |
| **経験ベース**   | 失敗から生まれた診断ルール（原則）         | 反省→検証成功時 |


> **図4**（論文 Figure 4）: 医師エージェントが症例ベース・経験ベースを RAG 検索して診断する流れ

**反省の流れ**（失敗時）：

```
1. 医師の診断・処方 vs 正解（シミュレータが保持）
2. LLM に「なぜ間違えたか」を問う
3. 診断ルール（テキスト）を生成
4. そのルールで同様のケースを再試行
5. 成功 → 経験ベースに追加 / 失敗 → 破棄
```

### 5-4. 経験ベースと RAG

推論時は、現在の症例に対して**医療症例ベース**と**経験ベース**の両方から類似項目を検索し、プロンプトに載せます。

- エンコーダ：論文では OpenAI `text-embedding-ada-002`
- 検索：コサイン類似度で上位 n 件の症例 + 上位 k 件のルール
- ローカル再現では `sentence-transformers` + FAISS 等で代替可能（Part 2）

固定の外部知識ベースだけを参照する RAG との違いは、**経験ベースが診察を重ねるごとに動的に増える**点です。最初は医学知識ベースだけ、失敗を重ねるほどルールが追加されていきます。

### 5-5. 正誤判定は誰がやるか（LLM vs シミュレータ）

ここが誤解されやすいポイントです。


| 役割              | 担当                    |
| --------------- | --------------------- |
| 診断・治療の決定        | **LLM（医師エージェント）**     |
| 治療の実行と結果（回復/悪化） | **シミュレータ（プログラムのルール）** |


例：肺炎の患者に抗生物質を処方すれば体温が下がる、間違った治療なら悪化する——といった**医学的原則**がシミュレータに埋め込まれています。LLM が「正しい」と判定するのではなく、**環境がフィードバック**を返します。

「結局ラベルがあるのでは？」という疑問はもっともです。正確には：

- **従来型**：症例単位の「症状→診断」ラベルが大量に必要
- **Agent Hospital**：「肺炎に抗生物質が効く」など**疾患・治療単位の原則**がシミュレータにあり、そこから無限に症例を生成する

## 6. 実験と結果

### 6-1. 仮想病院での精度向上

仮想世界では、医師エージェントを次の3タスクで評価しています。

1. **医学的検査の選択** — 適切な検査を選べるか
2. **診断** — 疾病を正しく特定できるか
3. **治療計画の推奨** — 適切な治療を提案できるか

21 の臨床診療科それぞれについて、20,000 人の患者で「進化」、200 人の別セットでテスト（訓練データとの重複なし）。

**例（循環器科 / GPT-3.5）**：リウマチ性心疾患の診断精度が **9% → 82%** に向上。

全臨床診療科で改善。検査タスクで最大 +76%、診断タスクで最大 +31%。進化後の平均性能は多くの診療科で 95% 超。

### 6-2. スケーリング則

**進化におけるスケーリング則** = 治療した患者数を増やすほど、医師エージェントは強くなり続けるか、という問い。

図5b（呼吸器科、50,000 人）の傾向：

- 初期精度 ~66%
- 最初の 10,000 人で急上昇（症例ベース・経験ベースが急速に成長）
- その後も改善は続くが、ペースは緩やかに

> **図5a / 5b**（論文 Figure 5）: 診療科別の精度改善と、患者数に対する精度曲線

### 6-3. MedQA とアライメント（論文側の「仮想→現実」）

**MedQA** は USMLE 形式の医学問題データセット。**評価用（テスト）**であり、MedAgent-Zero の学習には使っていません。

論文での **アライメント** = 「仮想病院で強くなった医師エージェントが、現実の MedQA でも強くなるか」（※ RLHF のアライメントとは別概念）。

- 図5b：仮想世界の診断精度 ↑ に伴い、MedQA（呼吸器疾患）の正答率も ↑ する傾向
- MedQA 全体（GPT-4o ベース）で **92.22%**、既存手法（MedAgents, CoT, Medprompt）を上回る


| 手法                | GPT-3.5   | GPT-4     | GPT-4o    | o1-preview |
| ----------------- | --------- | --------- | --------- | ---------- |
| CoT               | 57.06     | 76.83     | 84.45     | —          |
| Medprompt         | 66.30     | 84.45     | 89.24     | —          |
| MedAgents         | 66.30     | 84.45     | 89.24     | —          |
| **MedAgent-Zero** | **74.31** | **89.71** | **92.22** | **96.15**  |


興味深い点：MedAgent-Zero（GPT-4）が GPT-4o の Direct 手法を上回るケースも報告されています。

### 6-4. 限界と考察（論文側）

論文著者自身が挙げる制限：

- **ベース LLM は凍結** — モデル自体は進化しない
- **高レベルの治療計画**に留まる — 詳細な臨床判断の全てをカバーするわけではない
- **診療科間の相談**（会诊）がない
- 倫理・バイアス・説明責任の課題

本記事シリーズの Part 3 では、これらを「実臨床投入」の話ではなく、**RAG チャットへの設計転用**の文脈で整理します。

## 7. 論文を読んで残った疑問

論文を読みながらメモした FAQ です。7章だけの理解でなく、5章の補足としても読んでください。

### 7-1. 「ラベルなし」の正確な意味

**「症例単位の教師ラベルがない」** が正確です。

医学知識ベースやシミュレータの治療ルールには、医学的原則レベルの「正解情報」が入っています。Zero とは「人間が1万件の症例に診断ラベルを付けたデータセットで学習しない」という意味に近いです。

### 7-2. 医学知識ベースと正解データの違い


|     | 医学知識ベース         | 症例ラベル（従来型）    |
| --- | --------------- | ------------- |
| 内容  | 「肺炎とは何か、症状、治療法」 | 「患者A → 診断は肺炎」 |
| 用途  | 診断時の参考（RAG）     | 教師あり学習の正解     |
| 粒度  | 疾患・治療の一般知識      | 個別症例          |


シミュレータの回復/悪化判定は、知識ベースから導いた**ルール**で動きます。

### 7-3. 経験ベースの矛盾・スケール問題

経験ベースは**追加していくだけ**に見えるため、「矛盾するルールが溜まって精度が落ちないか」と気になります。

論文では詳細な整理機構は明示されていませんが、実験上は規模が増えても精度低下は見られていません。考えられるのは：

- ベクトル検索で**関連性の高い上位 k 件**だけを使う
- 失敗したルールは検証段階で**破棄**される
- プロンプトで矛盾時の優先順位を指示

ミニ版再現（Part 2）では、Append-only + 論理削除（`is_active = False`）など、自分なりの整理も試す予定です。
