2. 背景 — なぜ Agent Hospital か

Part 0 で触れたとおり、病院向け RAG チャットアプリを開発しています。本章では、なぜその延長として Agent Hospital 論文を深く読んだのかを整理します。

2-1. エージェント実装を検討する理由

RAG の外側にも足せる設計に興味を持ちました。複数ステップの処理、ツール呼び出し、状態管理、失敗から学ぶ仕組み——これらは RAG を置き換える話ではなく、既存の RAG に並べて載せられるレイヤーだと考えています。Agent Hospital は医療ドメインでの参考事例として目に留まりました。

2-2. RAG に加えて足せる拡張の考え方

RAG チャットのコアはそのまま活かしつつ、必要に応じて別ロジックを組み込めます。RAG は知識検索の部品として中心に置けるままです。

レイヤー例
RAG資料検索、類似症例・ルールの取得
ワークフローグラフ / ステートマシン
ツール (Tool Use)ファイル、API、シミュレータ
反省失敗分析 → ルール化
ルール前処理・後処理、ガードレール

MedAgent-Zero は RAG 的な経験ベースを使いますが、正誤判定はシミュレータが担います。RAG を残したまま別レイヤーを足す設計の参考になります。

2-3. 医療 LLM（Med-PaLM 等）の限界

論文が対比するのは、Med-PaLM のようにスクラッチから学習した、またはファインチューニングした医療 LLM です。膨大なコーパスでモデル重みを更新し医学知識を組み込む一方、病院ワークフロー全体を直接モデル化する設計ではありません。

2-4. スクラッチ・ファインチューニング vs 実践シミュレーション

スクラッチ / FT実践シミュレーション
何をするか大規模コーパスで事前学習 or FT仮想病院で診察（LLM 凍結）
強み広い医学知識・試験問題ワークフローに沿った経験蓄積
弱み実践プロセスの再現シミュレータ設計への依存

院内資料 Q&A 向けの RAG チャットと Agent Hospital は用途が違うだけです。後者から学べるのは、ワークフローや経験蓄積の足しものの設計思想です。

2-5. この論文に辿り着いた理由

  1. 病院向け RAG チャットを開発中
  2. エージェント化を次のステップとして検討
  3. RAG 以外のロジックも組み合わせられる設計を探していた
  4. Agent Hospital = 医療 × マルチエージェント × 経験蓄積
  5. 論文を読み、ミニ版を再現し、RAG チャットへの適応を整理（Part 2 / 3）

3. Agent Hospital の全体像

3-1. シミュラクラ（仮想病院）とは

シミュラクラとは現実を模倣した仮想環境のこと。患者・看護師・医師がすべて LLM エージェントで、仮想時間は現実より数桁速く進みます。

3-2. 登場するエージェント

種類役割
患者エージェント症状を訴え、検査・治療結果で回復/未回復
看護師エージェントトリアージ（症状を聞き取り、緊急度・診療科を振り分ける）、受付、検査結果の受け渡し
医師エージェント問診、検査指示、診断、処方

32 診療科・300 以上の疾病をカバー。医師 42 名・看護師 4 名（論文設定）。

3-3. 診察〜治療の流れ

発病 → トリアージ（症状聞き取り・振り分け） → 受付 → 診察 → 医学的検査 → 診断・処方 → 薬の調剤 → 回復期

回復期のフィードバックが重要です。患者が回復したかどうかがシミュレータから返り、正誤判定と進化のトリガーになります。医師には勤務外の読書（医学書から知識統合）イベントもあります。

4. SEAL — 提案パラダイム

SEAL（Simulacrum-based Evolutionary Agent Learning）は、シミュラクラ構築とエージェント進化の2フェーズから成ります。

① シミュラクラ構築 — LLM + 医学知識ベースで、人間のアノテーションなしに患者データ・症状・検査結果・経過を自動生成。

② エージェント進化 — 成功 → 医療症例ベースへ。失敗 → 反省して診断ルールを生成 → 経験ベースへ。次の診察で RAG 的に参照（MedAgent-Zero、5章）。

データ生成自己対戦仮想病院での診察
正誤判定勝敗患者の回復/未回復

Smallville が「仮想社会のエージェント」を示したのに対し、Agent Hospital は病院ワークフローに特化しています。

5. MedAgent-Zero — 進化の仕組み

5-1. 「Zero」の意味

Zero = 手動ラベル付き症例データを使わない。LLM は凍結し、経験はテキスト（症例ベース・経験ベース）で蓄積します。

従来型（ラベル有り）MedAgent-Zero
学習データ「症状→診断」ラベル数万件シミュレーション自動生成
モデルファインチューニングLLM 凍結、経験はテキストで蓄積

5-2. 患者エージェントの自動生成

疾病名から、基本情報 → 病歴 → 症状 → 検査報告を LLM + 知識ベースで生成。品質管理エージェントが整合性をチェックします。

5-4. 経験ベースと RAG

推論時に text-embedding-ada-002 + コサイン類似度で両ベースから類似項目を検索しプロンプトに載せます。経験ベースは診察を重ねるごとに動的に増える点が、固定の外部知識のみの RAG との違いです。

5-5. 正誤判定は誰がやるか

役割担当
診断・治療の決定LLM（医師エージェント）
回復/悪化の判定シミュレータ（プログラムのルール）

「ラベルなし」= 症例単位の教師ラベルがない。シミュレータには疾患・治療単位の医学的原則が埋め込まれています。

6. 実験と結果

6-1. 仮想病院での精度向上

3タスク（検査選択・診断・治療計画）で評価。21 診療科 × 20,000 人で進化、200 人でテスト。例：循環器科 / リウマチ性心疾患が 9% → 82%（GPT-3.5 ベース）。全臨床診療科で改善、進化後は多くで 95% 超。

6-2. スケーリング則

治療患者数を増やすほど精度が上がるか。呼吸器科 50,000 人では初期 ~66% から、最初の 10,000 人で急上昇し、その後も緩やかに改善が続きます。

図5a / 5b（論文 Figure 5）: 診療科別改善と患者数に対する精度曲線

6-3. MedQA とアライメント

MedQA（USMLE 形式）は評価専用。仮想精度 ↑ に伴い MedQA 正答率も ↑。GPT-4o ベースで 92.22%、MedAgents / Medprompt 等を上回る。

6-4. 限界と考察（論文側）

  • ベース LLM は凍結（モデル自体は進化しない）
  • 高レベルの治療計画に留まる
  • 診療科間の相談がない
  • 倫理・バイアス・説明責任の課題

7. 論文を読んで残った疑問

7-1. 「ラベルなし」の正確な意味

症例単位の教師ラベルがない。医学知識ベースとシミュレータの原則は使う。

7-2. 医学知識ベースと正解データの違い

知識ベース = 疾患の一般知識（診断の参考）。症例ラベル = 個別症例の正解（従来型教師データ）。回復/悪化はシミュレータのルールで判定。

7-3. 経験ベースの矛盾・スケール問題

ルールが追加され続けても論文実験では精度低下なし。上位 k 検索、失敗ルールの破棄、プロンプトでの矛盾指示などが考えられる。Part 2 では Append-only + 論理削除も試す予定。
